1. Архитектура
В качестве архитектурного стиля выбрал модульный монолит с четким разделением слоев и ответственности,
это позволит сохранить простоту разработки на этапе MVP но при этом заложить фундамент для масштабирования в будущем.
В основе лежит классический трехслойный подход с выделением слоев:
- Application слой, содержит use-case классы которые описывают сценарии использования и бизнес-логику
- Infrastructure слой, содержит адаптеры для взаимодействия с внешними системами (HTTP клиенты, база данных, HTML парсеры)
- Shared слой, содержит общие утилиты и контракты, которые могут использоваться в разных частях приложения
В ближайщей перспективе можно легко добавить слой с очередями для распределения задач между воркерами без изменения бизнес-логики.
В дальшейшем при росте нагрузки можно будет перейти на микросервисы выделив отдельные сервисы для пасеров, API, аналитики, и планировщика задач для обеспечения гибкости.
Для очередей можно использовать BullMQ c Redis, также при росте объемов можно будет добавить OLAP-слой для аналитики и отчетности использовав ClickHouse, в итоге получится связка OLTP PostgreSQL + OLAP ClickHouse для аналитики.

2. База данных
Буду описывать особенности таблиц а не полный список
 - Marketplace - нужна для разделения данных между регионами Amazon, уникальность по code
 - Product - для дедупликации выбран составной ключ (marketplaceId + asin), добавленны индексы для аналитических выборок на будущее
 [marketplaceId, reviewsCount] для аналитики по объему отзывов
 [marketplaceId, averageRating] для аналитики по рейтингу
 [brandId] - для выборок по бренду
 [lastReviewCrawlAt] отбор товаров для следующего прохода
 [lastSeenAt] выборки для контроля свежести данных по товарам
 - Сategory + ProductCategory - для построения дерева категорий и связи товаров с категориями, уникальность по marketplaceId + externalId
 Индексы:
 [categoryId, rankInCategory] для быстрой выборки товаров категории по позиции
 [marketplaceId, externalId] для быстрого поиска/фильтрации категории по внешнему идентификатору
 [marketplaceId, path] для выборки/фильтрации категорий по пути
 один товар может быть в нескольких категориях.
 - Seller + ProductSeller - для хранения информации о продавцах и их связи с товарами, уникальность по marketplaceId + externalSellerId
 один товар может предлагаться несколькими продавцами.
 - Brand - отдельная таблица для сохрания нормализванных названий брендов
 - CrawlRun + CrawlRunItem - фиксация каждого запуска парсера и логирование его метрик
 [productId, createdAt] - селект для выборки по истории проблем по конкретному товару
 [reviewId] - для выборки проблем по отзывам
 [status, startedAt] - для мониторинга запусков
 - SyncCheckpoint - сохранение чекпоинтов для синхронизации категорий и отзывов, привязка к categorySlug + category URL signature для корректного возобновления после сбоев
 Индексы:
   [scope, checkpointTime] - для выборок что давно не обновлялось
 - [productId] и [categoryId] - для быстрых выборок по товарам или категориям
 - Review - для хранения отзывов, уникальность по marketplaceId + externalReviewId, индексы для выборок по продукту и свежести данных
 Индексы
 [productId, reviewedAt] для запросов по типу 'последние отзывы по продукту'
 [productId, lastSeenAt] для контроля свежести данных и сервисных задач
В данном MVP осознанно выбрана PostgreSQL так как на данном этапе она обеспечивает достаточную производительность
Также уже реализованны некоторые индексы для обеспечения быстродействия при росте объемов данных, а в дальнейшем при необходимости можно будет перейти на более масштабируемые решения и добавить OLAP-слой для аналитики.
3. Сценарий работы
Для MVP версии выбран запуск скрипта через CLI parse:amazon с передачей параметров
Для category и  страниц с товарами используется HTTP + HTML парсинг, axios + cheerio,
был  применен retry/backoff и random delay для обхода защиты от ботов и обеспечения устойчивости, если
мы получаем signIn страницу или anitbot challenge то делается fallback на curl запрос
Для отзывов изначально также использовалась HTTP + HTML парсинг, но из-за защиты от ботов и signIn страниц был реализован
вариант с playwright, для него требуется json с сессией который можно получить c помощью команды
npm run review:login, сейчас есть сохраненная сессия поэтому можно без нее. На данный момент для тестового задания это оптимальный подход
но на продакшн версии нужно будет перейти на вариант с http + html парсинг + прокси для обхода защиты от ботов, так как использование браузера является
неэффективным на больших объемах.
Сценарий работы:
1. Запуск парсера через CLI с указанием marketplace и категории
2. Валидируется url категории 
3. В БД делается upsert в таблицы Marketplace и Category, создается запись в CrawlRun со статусом RUNNING
4. Парсер обходит страницы категорий, загружает HTML, извлекает данные товаров, дедуплицирует по asin и обновляет categoryCheckpoint для
продолжения с указанной точки в случае сбоев.
5. Сохранение карточек товаров в БД, для каждого asin делается upsert в Product и связь с категорией в ProductCategory
6. Далее запускается синхронизация деталей товаров ProductSyncUseCase загружается страница товара, парсятся brand seller price ratings и делается update в таблицах
Product, Brand, Seller, ProductSeller.
7. Запускается ReviewSyncUseCase, строится URL страниц с отзывами получается HTML через ReviewFetchService
есть 2 варианта http или playwright, парсятся отзывы и сохраняются в таблицу Review, также обновляется checkpoint для продолжения с указанной точки в случае сбоев
8. Для каждого Review есть механизм инкрементального изменения если записи нет то она создается, если изменился fingerprint то обновляется,
если изменился только lastSeenAt то обновляется только он, это позволяет отслеживать изменения в отзывах и контролировать свежесть данных.
9. После завершения парсинга категории обновляется статус CrawlRun на SUCCESS, если были ошибки то сохраняется информация об ошибках и статус FAIL, также сохраняются метрики по количеству обработанных товаров и отзывов для аналитики.
